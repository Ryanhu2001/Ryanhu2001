# Paper Reading: Dockerless — 无需执行环境的编码智能体验证器

> **论文信息**：Dockerless: Environment-Free Program Verifier for Coding Agents
> **arXiv**: 2606.28436
> **机构**：上海交通大学 + 抖音集团
> **领域**：软件工程 / 编码智能体 / 奖励建模
> **关键词**：程序验证、编码智能体、奖励模型、无环境训练

---

## 一、一句话总结

**Dockerless 是一个不需要 Docker 执行环境的代码补丁验证器**。它不靠跑单元测试来判断补丁对不对，而是让子智能体主动去代码库里"调查取证"，基于收集到的证据给出正确性评分。用它替代传统的执行式验证，可以让编码智能体的 SFT 筛选和 RL 奖励完全摆脱仓库级环境配置，大幅降低训练成本，同时效果几乎不打折。

---

## 二、为什么要做这件事？—— 背景与痛点

### 2.1 编码智能体训练的"验证瓶颈"

现在训练 SWE（软件工程）智能体有两个核心环节都依赖"程序验证器"：

- **SFT 阶段**：从大量 rollout 里筛选出正确解决问题的轨迹，用来做监督微调
- **RL 阶段**：给每个 rollout 打分作为奖励信号，驱动模型迭代优化

目前行业的标准做法是**执行式验证**：把补丁放到对应仓库的 Docker 镜像里跑单元测试，通过了就算正确。这很准，但代价极大。

### 2.2 执行式验证 vs Dockerless 对比

![执行式验证 vs Dockerless对比](https://aka.doubaocdn.com/s/1kCJ1wrdjl)

### 2.3 执行式验证的三大痛点

1. **工程成本极高**：每个仓库都要定制 Docker 镜像、解决依赖、找测试用例、写执行脚本。自动化流水线成功率也有限。
2. **覆盖范围有限**：大量真实仓库（私有、企业级、遗留代码）根本没有可复现的环境和完整测试集，执行验证直接不可用。
3. **长尾无法规模化**：现实中的代码仓库是"长尾分布"的，不可能为每个仓库都搭一套执行环境。

### 2.4 已有方案的不足

之前的无环境验证器基本就是让 LLM 直接对比候选补丁和参考补丁的文本差异，属于"浅层比对"。它不看代码库上下文，判断不了功能等价性——比如同样的修复逻辑，写法不一样就容易判错。

> 💡 **核心洞察**：判断补丁正确与否，关键不在于"长得像不像参考补丁"，而在于"能不能在代码库的上下文中真正解决问题"。

## 三、Dockerless 是怎么工作的？—— 核心方法

### 3.1 整体架构

![Dockerless两阶段验证架构](https://aka.doubaocdn.com/s/pXUr1wrdjh)

Dockerless 不走"文本相似度"的老路，而是模拟人类代码审查的思路：

> 先提出几个关键验证问题 → 分头去代码库里找答案 → 汇总证据后下结论

整个流程分为**两阶段**。

### 3.2 阶段一：问题生成 + 并行探索

**输入**：问题描述 `x` + 参考补丁 `y_ref` + 候选补丁 `y`

1. **生成 K 个验证问题**。比如：
   - 修复应该在仓库的哪个位置生效？
   - 修改后的代码应该实现什么功能？
   - 哪些测试或断言可以验证正确性？
   - 修改会不会影响仓库其他模块？

2. **并行派出 K 个子智能体**，每个子智能体带着一个问题去代码库中探索。它们使用 `find`、`grep`、`rg` 等只读工具检索代码，返回**带证据的答案**。

> 为什么要并行？因为探索过程是 IO 密集型的，并行可以大幅缩短总耗时。

### 3.3 阶段二：证据聚合 + 最终判决

收集到 K 组 (问题, 答案) 证据对之后，判决模型综合所有信息，输出一个二分类 token（0=错误，1=正确）。

推理时用两个 token 的 logit 计算出一个 0~1 之间的连续分数：

$$r_\phi(x,y) = \frac{\exp(\ell_1)}{\exp(\ell_0) + \exp(\ell_1)}$$

这个分数就可以直接用作 SFT 筛选阈值或 RL 奖励。

### 3.4 训练方式：拒绝采样

Dockerless 的训练很有意思，用的是**拒绝采样（Rejection Sampling）**：

1. 先用一个教师模型生成完整的"问题-探索-判决"轨迹
2. 把最终判决结果和真实执行结果（跑单元测试的 ground truth）做对比
3. **只保留判决正确的轨迹**用来训练
4. 正负样本比例做截断，缓解类别不平衡

这样训练出来的模型学到的是"正确的推理链条"，而不是靠运气蒙对。

> 🔑 **技术亮点**：问题生成、子智能体探索、最终判决**共享同一个骨干模型**，一起端到端训练。

## 四、完整的无环境训练流水线

![无环境编码智能体训练流程](https://aka.doubaocdn.com/s/6gY31wrdjl)

有了 Dockerless 之后，整个编码智能体的后训练都可以摆脱 per-repository Docker：

### 4.1 无环境 SFT（RFT）

- 在一个极简 Ubuntu 基础镜像里收集大量 rollout（不用装每个仓库的依赖）
- 用 Dockerless 给每个补丁打分
- 取分数最高的 Top-K 轨迹做监督微调

### 4.2 无环境 RL

- 同样在极简环境里收集 rollout
- Dockerless 的分数直接作为奖励信号
- 用 GRPO 等 RL 算法优化模型

> 整个过程不需要为任何一个仓库单独搭建 Docker 环境，这意味着训练可以轻松扩展到成千上万的长尾仓库。

## 五、实验结果怎么样？

### 5.1 验证器本身的性能

在 SWE-bench 上对比不同验证器：

| 方法 | 类型 | AUC |
|------|------|-----|
| 基于文本相似度的基线 | 无环境 | ~0.70 |
| **Dockerless** | **无环境** | **0.843** |
| 执行式验证（Oracle） | 有环境 | 1.0 |

- Dockerless 比传统无环境验证器**提升了 14.3 个百分点的 AUC**
- 虽然还达不到执行式验证的 100% 准确率，但已经相当可用

### 5.2 用于智能体训练的效果

用 Dockerless 替代执行式验证来做 SFT 筛选和 RL 奖励，在 SWE-bench 上的表现：

- **无环境 SFT**：性能接近基于执行验证的 SFT，几乎没有损失
- **无环境 RL**：同样能带来显著的性能提升
- **整体结论**：用 Dockerless 训练出来的智能体，和用执行式验证训练的效果差不多，但成本低得多

## 六、意义与价值

### 6.1 对行业的影响

1. **大幅降低训练门槛**：小团队也能训高质量编码智能体，不用搭复杂的 per-repo 环境
2. **解锁长尾数据**：大量没有测试、没有可复现环境的真实仓库，现在也能用来训练了
3. **训练效率提升**：不用等 Docker 启动、不用装依赖，rollout 收集速度快很多

### 6.2 技术思路的启发

Dockerless 代表了一个很有意思的方向：**用智能体能力来替代执行环境**。

- 传统思路：验证 = 执行 + 观察结果
- 新思路：验证 = 推理 + 探索 + 证据判断

这和人类代码审查的方式更像——资深工程师 review 代码时，也不是每段都要跑一遍，而是靠对代码逻辑的理解和上下文的把握来判断对错。

### 6.3 局限性

- 目前还是不如执行式验证准确，极端复杂的场景可能判错
- 依赖模型对代码的理解能力，模型越强验证器越准
- 对于需要实际运行才能发现的问题（比如性能、并发 bug），纯静态推理可能无能为力

## 七、总结

| 维度 | 要点 |
|------|------|
| **问题** | 编码智能体训练依赖执行式验证，成本高、覆盖有限 |
| **方法** | 两阶段架构：问题生成+并行探索 → 证据聚合+判决 |
| **核心创新** | 用子智能体主动探索代码库收集证据，而非浅层文本比对 |
| **效果** | AUC 提升 14.3 个点，训练效果媲美执行式验证 |
| **价值** | 让编码智能体训练摆脱 per-repository Docker，大幅降本增效 |

> 🎯 **一句话评价**：Dockerless 用"智能体推理"替代"环境执行"，是编码智能体训练基础设施的一次重要范式升级。它不一定能完全取代执行验证，但会成为大规模、低成本训练的重要拼图。
